## 2. 开发约定与规范

### 2.1 页面注册约定

- 使用 `@Page("pageName", supportInLocal = true)` 注解注册页面
- 所有页面继承 `BasePager` (它继承 Kuikly 的 `ComposeContainer`)
- 在 `willInit()` 中调用 `setContent { ... }` 设置 Compose UI
- 页面名与 `render.renderView({ pageName: "xxx" })` 中的名称对应
- 主包占位页面 `pages/index/index` 必须存在（小程序要求）

```kotlin
// 示例: shared/src/commonMain/.../MainPage.kt
@Page("main", supportInLocal = true)
internal class MainPage : BasePager() {
    override fun willInit() {
        super.willInit()
        setContent { MainContent() }
    }
}
```

### 2.2 平台差异处理约定

使用 Kotlin `expect/actual` 机制处理平台差异，**不鼓励运行时 polyfill**：

| 差异点 | expect 定义位置 | actual 实现位置 |
|--------|----------------|----------------|
| Base URL | `commonMain/.../data/BaseUrl.kt` | 各平台 `xxxMain/.../BaseUrlOverride.kt` |
| 页面跳转 | `commonMain/.../base/OpenWebDetail.kt` | 各平台 `xxxMain/.../base/OpenWebDetail.kt` |
| 数据库驱动 | `commonMain/.../data/DatabaseDriverFactory.kt` | 各平台实现 |

```kotlin
// commonMain - 声明
expect fun openWebDetail(url: String, title: String)
expect fun getBaseUrl(): String

// jsMain - 小程序实现
actual fun openWebDetail(url: String, title: String) {
    js("wx.navigateTo({url: '/packageA/pages/webview/webview?url=' + encodeURIComponent(url)})")
}
actual fun getBaseUrl(): String = "https://www.wanandroid.com/"
```

### 2.3 网络层约定

- **HTTP 客户端**: Ktor (`HttpClient`) + Ktorfit (声明式 API)
- **序列化**: kotlinx.serialization (`@Serializable`)
- **API 定义**: 接口 + `@Serializable` 数据类，集中在 `data/WanApi.kt`
- **响应格式**: 统一 `ApiBaseResponse<T>` (errorCode, errorMsg, data)
- **单例配置**: `KtorfitInstance` 提供 `httpClient` 和 `api` 实例

```kotlin
interface WanApiService {
    suspend fun getHomeArticles(page: Int): ApiBaseResponse<ArticleListData>
    suspend fun login(username: String, password: String): ApiBaseResponse<JsonElement>
    // ...
}
```

### 2.4 构建配置约定

#### shared/build.gradle.kts (KMP 业务模块)

```kotlin
js(IR) {
    browser {
        webpackTask {
            mainOutputFileName.set("nativevue2.js")
            webpackConfigApplier {
                // 设置 publicPath 避免 document 不存在时报错
                // config.output.publicPath = ''
            }
        }
        commonWebpackConfig {
            output?.library = null    // 不导出全局对象
            devtool = "source-map"    // 独立 source map 文件
        }
    }
    binaries.executable()             // 打包为可执行 JS
}
```

**polyfill 注入机制** (关键配置):
- `injectDocumentMock()` 函数在 webpack 构建完成后执行
- 将 polyfill 代码**前置插入**到 `nativevue2.js` 文件最顶部
- 注入后删除 source map 引用 (行号已变化)
- 包含: window.open、TextEncoder/TextDecoder、document mock、AbortController、fetch、`__decodeUTF8()`

#### miniApp/build.gradle.kts (小程序渲染引擎)

```kotlin
webpackTask {
    outputFileName = "miniApp.js"
    webpackConfigApplier {
        config.target = 'node'  // ★ 必须为 'node'，小程序有 global 无 window
    }
}
```

#### 版本管理

- 统一在 `buildSrc/Version.kt` 管理
- Kuikly 版本格式: `${shortVersion}-${kotlinVersion}` (例: `2.7.0-2.1.21`)
- Maven 仓库: 腾讯镜像 `mirrors.tencent.com`

### 2.5 小程序运行时约定

#### 初始化顺序 (严格)

```
app.js (主包, 最先执行)
  ├─ 全局 polyfill (TextEncoder/TextDecoder)     ← 使用 global.xxx
  └─ 预加载 require('./lib/miniApp.js')           ← 利用主包 CommonJS 上下文
     └─ 缓存到 global.__kuiklyRender

main.js (分包, 用户进入分包时执行)
  ├─ require('../../business/nativevue2.js')      ← 加载业务代码
  ├─ render = global.__kuiklyRender               ← 使用缓存 (避免跨包 require 异常)
  ├─ global.com = business.com                    ← 挂载业务入口
  ├─ global.callKotlinMethod = business.callKotlinMethod
  ├─ render.initApp()                             ← 初始化渲染引擎 (必须在业务代码之后)
  └─ render.renderView({ pageName: "main" })     ← 渲染页面
```

#### 分包加载规则

- 主包大小限制: 2MB → 业务代码放在分包
- miniApp.js (渲染引擎) 放在主包 `lib/` (多页面共用，不能放分包)
- nativevue2.js (业务代码) 放在分包 `packageA/business/`
- 路径计算以页面所在目录为基准

## 3. 开发流程

### 3.1 新增页面流程

1. **Kotlin 层**: 在 `shared/src/commonMain/` 对应模块目录创建页面类
   ```kotlin
   @Page("detail", supportInLocal = true)
   internal class DetailPage : BasePager() {
       override fun willInit() {
           super.willInit()
           setContent { DetailContent() }
       }
   }
   ```

2. **平台适配** (如需要): 在对应平台源码集写 `actual` 实现

3. **小程序端**: 无需手动创建 wxml/json/js 文件 — Kuikly KSP 编译器自动生成
   - `@Page` 注解被 KSP 处理，自动生成 `render.renderView({ pageName: "detail" })` 的调用代码
   - 渲染引擎通过 `base.wxml` 模板动态渲染 Compose UI

4. **构建**: 运行 `./gradlew :shared:jsBrowserProductionWebpack` + `./gradlew :miniApp:jsMiniAppProductionWebpack`

### 3.2 新增网络接口流程

1. **定义数据类**: 在 `WanApi.kt` 中添加 `@Serializable` 数据类
2. **定义接口方法**: 在 `WanApiService` 接口中添加 `suspend fun`
3. **实现接口**: Ktorfit 自动根据注解生成实现 (无需手动实现)
4. **调用**: 通过 `KtorfitInstance.api.xxx()` 调用

### 3.3 构建与部署流程

```bash
# 1. 构建小程序渲染引擎 (miniApp.js)
./gradlew :miniApp:jsBrowserProductionWebpack
# → 自动同步到 dist/lib/miniApp.js

# 2. 构建业务代码 (nativevue2.js)
./gradlew :shared:jsBrowserProductionWebpack

# 3. 一次性构建 (推荐)
./gradlew :miniApp:jsMiniAppProductionWebpack
# → 构建 shared + miniApp → 拷贝 nativevue2.js 到 dist/packageA/business/

# 4. polyfill 自动注入
# shared/build.gradle.kts 中 injectDocumentMock() 在 webpack 构建后自动执行
```

**手动维护的文件** (构建不会覆盖):
- `dist/app.js` — 主包入口 (polyfill + 预加载逻辑)
- `dist/packageA/pages/main/main.js` — 分包入口 (require + 初始化逻辑)
- `dist/app.json` — 小程序配置 (页面注册、分包声明)
- `dist/packageA/pages/*/` — 页面的 wxml/json/js 文件

### 3.4 调试流程

1. **先用 DevTools 验证**: 微信开发者工具基础功能验证
2. **必须真机调试**: DevTools 基于 Chromium，API 差异巨大，真机是最终检验标准
3. **保留 build 产物**: 构建目录的原始产物是恢复的基线
4. **修改 dist 文件**: 直接修改 dist/ 下文件可快速验证，但重新构建会覆盖
5. **同步构建配置**: 验证通过后，将修改同步到 build.gradle.kts 中

## 4. 真机调试问题全记录

> 以下问题全部在真机调试中发现，DevTools 表现正常。

### 4.1 路径解析失败：custom-wrapper.json 找不到

- **错误**: `Error: [summer-compiler] Couldn't found the '../../custom-wrapper.json' file relative to 'packageA/pages/main/main'`
- **文件**: `main.json`, `main.wxml`
- **根因**: 页面在 `packageA/pages/main/` 目录下，`../../` 只能回退到 `packageA/`，到不了项目根目录
- **修复**: 路径多加一级 `../`
  - `main.json`: `"../../custom-wrapper"` → `"../../../custom-wrapper"`
  - `main.wxml`: `../../base.wxml` → `../../../base.wxml`

### 4.2 模块未定义：nativevue2.js require 路径错误

- **错误**: `module 'packageA/pages/business/nativevue2.js' is not defined`
- **文件**: `main.js`
- **根因**: `main.js` 在 `packageA/pages/main/`，require 相对路径与实际文件位置不匹配
- **修复**:
  - `require('../business/nativevue2.js')` → `require('../../business/nativevue2.js')`
  - `require('../../lib/miniApp.js')` → `require('../../../lib/miniApp.js')`

### 4.3 重复 Page 注册

- **错误**: `Please do not register multiple Pages in packageA/pages/main/main.js`
- **文件**: `main.js`
- **根因**: 手动调用了 `Page({...})`，而 `render.renderView()` 内部也会注册 Page
- **修复**: 移除手动 `Page()` 调用，仅保留 `render.renderView({ pageName: "main" })`

### 4.4 初始化顺序错误：乱码 + 菜单失效

- **错误**: 真机文字乱码，底部菜单点击切换不了
- **文件**: `app.js` → `main.js`
- **根因**: `app.js` (主包) 中调用了 `render.initApp()`，此时 `nativevue2.js` 尚未加载，`global.com` 不存在
- **修复**:
  - 从 `app.js` 移除 `render.initApp()`
  - 在 `main.js` 中 require 业务代码和渲染引擎之后才调用 `render.initApp()`
  - 严格顺序: `nativevue2.js` 加载 → `global.com` 挂载 → `initApp()` → `renderView()`

### 4.5 TextEncoder/TextDecoder 不可用

- **错误**: `ReferenceError: Can't find variable: TextEncoder` (miniApp.js 内部)
- **文件**: `miniApp.js` (Kuikly 渲染引擎), `nativevue2.js` (Ktor 依赖)
- **根因**: 微信小程序真机 JS 引擎不包含标准 Web API (TextEncoder/TextDecoder)。DevTools 基于 Chromium 所以有
- **深层原因**: webpack UMD 闭包内部通过**作用域链**查找变量，不走 `global` 对象。因此 `global.TextEncoder = ...` 不生效
- **修复**:
  - **nativevue2.js**: 在 `shared/build.gradle.kts` 的 `injectDocumentMock()` 中注入 polyfill 到 webpack 产物顶部
  - **miniApp.js**: 在文件顶部用 `var TextEncoder` 声明 + polyfill（同一文件作用域，闭包可通过作用域链访问）
  - **关键点**: 必须用 `var` 声明（不是 `global.xxx`），使其处于文件作用域

```javascript
// 正确: var 声明在文件作用域，webpack IIFE 内部可通过作用域链访问
var TextEncoder;
if (typeof TextEncoder === "undefined") {
  TextEncoder = function() { this.encoding = "utf-8"; };
  TextEncoder.prototype.encode = function(str) { /* ... */ };
}

// 错误: global 对象赋值，闭包内部作用域链找不到
global.TextEncoder = function() { /* ... */ };
```

### 4.6 window.open 未定义

- **错误**: `TypeError: Cannot read property 'open' of undefined (window.open)`
- **文件**: `nativevue2.js` (Kotlin 编译产物中的 `window.open(t, "_blank")`)
- **根因**: 微信小程序没有 `window` 对象。即使添加 `globalThis.window` polyfill，webpack IIFE 闭包内仍无法访问
- **修复**:
  - **永久方案**: 修改 Kotlin 源码，使用 `expect/actual` 机制
    ```kotlin
    // commonMain
    expect fun openWebDetail(url: String, title: String)
    // jsMain
    actual fun openWebDetail(url: String, title: String) {
        js("wx.navigateTo({url: '/packageA/pages/webview/webview?url=' + encodeURIComponent(url)})")
    }
    ```
  - **配套**: 新建 `packageA/pages/webview/` 页面，使用 `<web-view>` 组件
  - 在 `app.json` 的 `subPackages` 中注册 webview 页面

### 4.7 网络响应中文乱码

- **错误**: `wx.request` 返回中文乱码，所有依赖网络的数据页面加载失败
- **文件**: `nativevue2.js` (fetch polyfill → Ktor HTTP 客户端)
- **根因**:
  1. `wx.request` 的 `responseType: 'text'` 在真机对 UTF-8 解码有 bug
  2. `TextDecoder` 在 `wx.request` 回调上下文中不可访问（即使 `globalThis.TextDecoder` 已定义）
- **修复**:
  - `responseType` 改为 `'arraybuffer'`，避免微信内置文本解码
  - 使用纯 JS 内联函数 `__decodeUTF8()` 手动解码 UTF-8，零外部依赖
  - `__decodeUTF8()` 定义在文件顶层（非闭包内），任何上下文均可调用

```javascript
// 纯 JS UTF-8 解码，零依赖
function __decodeUTF8(uint8) {
    var r = '', i = 0;
    while (i < uint8.length) {
        var b = uint8[i++];
        if (b < 0x80) r += String.fromCharCode(b);
        else if ((b & 0xE0) === 0xC0) {
            var b2 = uint8[i++];
            r += String.fromCharCode(((b & 0x1F) << 6) | (b2 & 0x3F));
        }
        // ... 3字节、4字节 UTF-8 处理
    }
    return r;
}
```

### 4.8 分包跨包 require 导致 UMD exports 丢失

- **错误**: `i.initApp is not a function. (In 'i.initApp()', 'i.initApp' is undefined)`
- **文件**: `main.js` → `require('../../../lib/miniApp.js')`
- **根因**: `miniApp.js` 是 webpack UMD 格式 bundle。当从分包跨包 require 主包模块时，微信小程序运行时对分包 JS 上下文的 `module`/`exports` 处理存在差异，导致 UMD 模块的 `module.exports` 未被正确赋值，require 返回空对象
- **修复**:
  - 在 `app.js` (主包) 中预先 `require('./lib/miniApp.js')`，利用主包完整的 CommonJS 上下文正确初始化 UMD 模块
  - 缓存到 `global.__kuiklyRender`
  - `main.js` 中优先使用缓存: `var render = global.__kuiklyRender || require('../../../lib/miniApp.js')`
  - 注意: `app.js` 只预加载，不调用 `initApp()`（需等 nativevue2.js 加载完毕）

## 5. 框架易错点与坑

### 5.1 DevTools vs 真机环境差异

| API | DevTools | 真机 | 影响范围 |
|-----|----------|------|----------|
| `TextEncoder` / `TextDecoder` | ✅ | ❌ | Ktor HTTP、编码 |
| `window` | ✅ | ❌ | 页面跳转 |
| `document` | ✅ | ❌ | webpack runtime |
| `fetch` | ✅ | ❌ | HTTP 客户端 |
| `AbortController` | ✅ | ❌ | 请求取消 |
| `global` | ✅ | ✅ | 全局对象 (小程序提供) |
| `Uint8Array` | ✅ | ✅ | 二进制处理 |

**铁律**: 真机调试是必须环节，DevTools 通过 ≠ 真机通过。

### 5.2 webpack IIFE 闭包作用域

webpack 打包的 JS 文件是 IIFE (立即执行函数表达式) 闭包:

```javascript
// webpack UMD 格式
!function(t, n) {
    // ... webpack 内部模块系统
    // 内部引用 TextEncoder → 走作用域链 → 文件作用域 → ❌ 不是 global
}(globalThis, () => { /* 模块代码 */ });
```

**关键规则**:
- `global.TextEncoder = ...` → ❌ 闭包内找不到
- `var TextEncoder = ...` (在文件顶部) → ✅ 闭包可通过作用域链找到
- 内联函数 (如 `__decodeUTF8`) → ✅ 最安全，零依赖

### 5.3 分包跨包 require 陷阱

微信小程序的模块系统:
- **主包**: `module`/`exports` 完整可用
- **分包**: 跨包 require 时 `module.exports` 可能被丢弃

**影响**: UMD 格式的 webpack bundle 依赖 `module.exports` 导出。跨包 require 可能拿到空对象。

**解决方案**: 在主包 `app.js` 中预先 require 并缓存。

### 5.4 wx.request responseType 陷阱

| responseType | DevTools | 真机 | 说明 |
|-------------|----------|------|------|
| `'text'` | ✅ 正常 | ❌ UTF-8 乱码 | 真机文本解码有 bug |
| `'arraybuffer'` | ✅ 正常 | ✅ 正常 | 需手动解码 UTF-8 |

**必须使用 `arraybuffer` + 手动 UTF-8 解码。**

### 5.5 shell 操作二进制 JS 文件

使用 `sed`、`cat` 等 shell 命令处理包含 `&0x` 等特殊字符的 JS 文件时，shell 会将 `&` 解释为后台运行或参数引用，导致内容被破坏。

```bash
# ❌ 危险: sed 中 &0x3f 被截断为 63
sed -i '' "1s/^/${POLYFILL}/" miniApp.js

# ✅ 安全: 使用 Python 处理
python3 -c "
with open('miniApp.js', 'r') as f: content = f.read()
with open('miniApp.js', 'w') as f: f.write(polyfill + content)
"
```
